# 分层软件开发 PRD 生成流程设计文档

**版本:** 1.0.0  
**日期:** 2026-05-17  
**状态:** 待评审

---

## 1. 概述

### 1.1 设计目标

本文档定义一个**可复用的 PRD（Product Requirements Document）生成流程**，服务于分层软件开发方法论。该流程的核心特性：

- **层级透明：** 除"是否为顶层"外，不区分模块级、组件级等具体层级概念
- **通用模板：** 所有层级使用同一套 PRD 模板结构（五大部分 + 三相格式）
- **独立调用：** 每次调用生成一份 PRD，两次调用之间无状态关联
- **人机协作：** 顶层通过结构化对话引导用户，后续层基于上层文档自动预填充

### 1.2 背景

在分层软件开发中，每一层都遵循"规格 → 设计 → 验证"的元工作流循环。PRD 生成是这一循环的第一步，负责将模糊的用户意图或上层约束转化为结构化的需求规格文档。

PRD 生成流程本身**不涉及**测试用例生成、架构设计、代码实现等后续步骤，仅聚焦于产出符合标准模板的需求文档。

---

## 2. 核心概念

### 2.1 两种工作模式

流程在每次调用时，通过自动判定选择以下两种模式之一：

| 模式 | 触发条件 | 核心行为 |
|:---|:---|:---|
| **Root 模式** | 无上层 PRD，或用户声明项目开端 | 通过多轮结构化对话完整收集 PRD 五大部分 |
| **Derive 模式** | 提供上层 PRD + 上层架构设计 + 目标模块 | 基于上层文档自动预填充，用户审阅修正 |

### 2.2 PRD 模板结构

所有层级统一使用以下五大部分结构：

| 部分 | 核心内容 | 撰写目标 | 篇幅占比 |
|:---|:---|:---|:---|
| **元信息 Frontmatter** | 文档 ID、版本、作者、状态、优先级、追踪链接 | 确保文档可追溯、可管理 | 5-10% |
| **问题陈述 Problem Statement** | 目标用户、痛点描述、机会窗口 | 建立需求的合理性和紧迫性 | 10-15% |
| **需求规格 Requirements** | 功能/非功能需求列表、MoSCoW 优先级 | 精确描述系统行为和质量属性 | 40-50% |
| **验收标准 Acceptance** | Gherkin 场景、边界条件、错误路径 | 定义"完成"的客观判定标准 | 20-25% |
| **成功指标 Success Metrics** | 业务指标、技术指标、用户指标 | 验证需求交付后的实际价值 | 10-15% |

### 2.3 三相文档格式

PRD 文档采用三相格式输出：

- **YAML frontmatter：** 承载结构化元数据（文档身份、状态、追踪链接、接口摘要）
- **Markdown body：** 承载人类可读的设计正文（五大部分）
- **Gherkin 场景：** 承载可执行的验收标准（Given-When-Then 格式）

## 3. 流程架构

### 3.1 入口与模式判定

流程以用户输入为起点，自动判定使用哪种模式：

```
[用户输入]
    │
    ├─ 情况 A: 自然语言描述（"我要做一个电商平台..."）
    ├─ 情况 B: 明确声明项目开端（"这是一个新项目..."）
    ├─ 情况 C: 上传上层 PRD + 架构设计 + 指定目标模块
    └─ 情况 D: 模糊输入（"帮我写个 PRD"）
    │
    ▼
[P0. 模式判定]
```

**判定规则（优先级从高到低）：**

1. **用户明确声明"项目开端" → Root 模式**（最高优先级，覆盖其他条件）
2. 同时满足以下三项 → Derive 模式：
   - 提供上层 PRD 文档
   - 提供上层架构设计文档
   - 明确指定目标模块名称
3. 以上条件不满足 → Root 模式

**情况 D 的处理：**
系统主动询问用户确认层级：
> "请确认这是项目的最顶层 PRD，还是基于已有 PRD 的下一层细化？"

## 4. Root 模式详细设计

### 4.1 适用场景

- 项目的最顶层 PRD（系统级需求）
- 用户只有一个初步想法，没有上层约束文档
- 用户明确声明"这是一个新项目/新功能的开端"

### 4.2 对话阶段

系统按五大部分顺序引导，每个阶段有明确的进入/退出条件：

| 阶段 | 目标 | 退出条件 |
|:---|:---|:---|
| P1. Frontmatter | 收集元信息 | 用户确认字段完整 |
| P2. Problem Statement | 理解业务背景 | 能回答"为什么做这个系统" |
| P3. Requirements | 收集功能/非功能需求 | 需求条目覆盖核心场景，MoSCoW 分类完成 |
| P4. Acceptance | 将需求转化为 Gherkin | 每条 Must-Have 需求至少对应 1 个 Gherkin 场景 |
| P5. Success Metrics | 定义验收指标 | 指标可量化、可追踪 |

### 4.3 P3. Requirements — 发散后收敛策略

**Step 1 — 发散收集：**
> "请描述这个系统应该具备哪些核心功能？用简短的一句话描述每个功能点。"

**Step 2 — 分类标注：**
系统自动应用 MoSCoW 分类，向用户确认优先级。

**Step 3 — 逐条精化（对每条 Must-Have）：**
> "关于'用户可以注册和登录'，请补充：
> 1. 支持哪些注册方式？
> 2. 登录后需要持久化会话吗？
> 3. 密码需要满足什么复杂度要求？"

**Step 4 — 非功能需求补充：**
> "请为整个系统定义非功能需求：并发用户数、响应时间要求、可用性目标..."

### 4.4 P4. Acceptance — Gherkin 引导生成

系统**不要求用户直接写 Gherkin**，而是引导用户描述场景，系统自动转化：

> "请描述一个用户成功注册的场景：
> - 用户当前处于什么状态？
> - 用户做了什么操作？
> - 系统应该有什么反应？"

系统实时转化为：
```gherkin
Scenario: 用户通过邮箱成功注册
  Given 用户未登录且访问注册页面
  When 用户输入有效邮箱 "user@example.com"
  And 用户输入符合复杂度要求的密码
  And 用户点击注册按钮
  Then 系统创建未验证用户账户
  And 系统发送验证邮件至该邮箱
```

用户确认或修改。

## 5. Derive 模式详细设计

### 5.1 适用场景

- 上层 PRD 和上层架构设计已完成
- 用户需要为架构设计中的某个具体模块生成 PRD
- 模块边界已由架构设计定义清楚

### 5.2 输入要求

| 输入项 | 是否必须 | 说明 |
|:---|:---|:---|
| `parent_prd` | 是 | 上层 PRD 文档内容或路径 |
| `parent_architecture` | 是 | 上层架构设计文档内容或路径 |
| `target_module` | 是 | 目标模块名称，须存在于架构设计的模块列表中 |

### 5.3 前置步骤：上下文解析

系统自动执行以下解析：

1. **校验：** 确认 `target_module` 存在于 `parent_architecture` 的模块列表中
2. **提取模块边界：**
   - 接口定义（OpenAPI 路径、请求/响应 Schema）
   - 依赖关系（本模块依赖哪些外部服务/模块）
   - 职责描述（模块一句话职责）
3. **提取相关需求：**
   - 从 `parent_prd` 提取与该模块相关的需求条目
   - 提取相关的 Gherkin 场景
   - 提取相关的非功能约束
4. **生成上下文摘要**展示给用户确认

### 5.4 各阶段策略（与 Root 模式对比）

| 阶段 | Root 模式 | Derive 模式 |
|:---|:---|:---|
| Frontmatter | 逐字段询问 | 自动生成，用户确认 |
| Problem Statement | 引导用户描述业务背景 | 基于上层需求提炼初稿，用户精化 |
| Requirements | 发散收集 → 分类 → 精化 | 继承上层相关需求 → 拆分细化 |
| Acceptance | 引导描述 → 转 Gherkin | 继承上层场景 → 补充内部场景 |
| Success Metrics | 引导定义指标 | 继承上层指标 → 分解模块级阈值 |

### 5.5 D3. Requirements — 上层需求拆分示例

**上层 PRD 条目：**
```markdown
- [REQ-005] 系统应支持多种支付方式（信用卡、支付宝、微信）
  - 优先级: Must Have
```

**系统拆分为模块级需求：**
```markdown
- [REQ-005-1] payment_gateway 应提供统一的支付发起接口
  - 支持参数: payment_method ∈ [credit_card, alipay, wechat]
  - parent_req: REQ-005
  - 优先级: Must Have

- [REQ-005-2] payment_gateway 应封装各渠道的差异化逻辑
  - Stripe: 使用 Stripe SDK v12+
  - 支付宝: 使用支付宝当面付接口
  - 微信: 使用微信支付 Native 接口
  - parent_req: REQ-005
  - 优先级: Must Have
```

用户审阅后，系统继续追问模块内部功能：
> "除了支付发起，支付网关内部还需要哪些功能？支付状态查询？回调处理？失败重试？对账？"

### 5.6 D4. Acceptance — 接口场景预填充

基于上层架构设计中的接口定义，系统**自动生成对外接口的 Gherkin 场景**：

上层架构接口定义：
```yaml
POST /api/v1/payments
request:
  order_id: string
  amount: number
  payment_method: enum[credit_card, alipay, wechat]
response:
  200: { payment_id, status: pending|success|failed }
  400: { error_code, message }
```

系统自动生成：
```gherkin
Scenario: 通过信用卡成功发起支付
  Given 订单模块调用 payment_gateway 的支付接口
  And 订单 "ORD-001" 金额 199.00 元
  When 使用信用卡方式发起支付
  Then payment_gateway 调用 Stripe 创建支付意图
  And 返回状态 200 且 payment_status 为 "pending"

Scenario: 使用不支持的支付方式
  Given 订单模块调用支付接口
  When 使用 "bitcoin" 作为支付方式
  Then 返回状态 400 且 error_code 为 "UNSUPPORTED_PAYMENT_METHOD"
```

用户补充模块内部场景（如第三方服务异常处理）。

### 5.7 信息隐藏原则

Derive 模式严格遵循信息隐藏原则：

- **关注当前模块内部：** 功能细节、算法逻辑、内部状态
- **关注当前模块对外契约：** 接口签名、输入输出格式（来自上层架构）
- **不关注同层其他模块：** 其他模块的内部实现与本模块 PRD 无关

## 6. 输出格式规范

### 6.1 文件命名

```markdown
# Root 层
{project_name}_prd_v{version}.md
# 示例: ecommerce_platform_prd_v1.0.md

# Derive 层
{parent_id}_{module_name}_prd_v{version}.md
# 示例: ecommerce_platform_v1.0_payment_gateway_prd_v1.0.md
```

### 6.2 文档结构模板

```markdown
---
doc_id: "ECOMMERCE-PLATFORM-v1.0"
version: "1.0.0"
layer: "root"                    # root | derive
parent_doc: null                 # 上级 PRD 的 doc_id
author: "Claude"
status: "draft"                  # draft | review | approved
priority: "P0"
created_at: "2026-05-17"
tags: ["ecommerce", "platform"]
---

# Problem Statement

## 目标用户
[描述]

## 痛点描述
[描述]

## 机会窗口
[描述]

---

# Requirements

## 功能需求

### Must Have
- [REQ-001] 系统应...
  - 来源: [业务方/技术债务/合规]
  - parent_req: []      # 上层需求 ID（root 层为空）

### Should Have
- [REQ-002] ...

### Could Have
- ...

## 非功能需求
- [NFR-001] 性能: 关键接口 P99 延迟 ≤ 200ms
- [NFR-002] 可用性: 系统可用性 ≥ 99.9%

---

# Acceptance

```gherkin
Feature: 用户注册
  @REQ-001 @critical
  Scenario: 通过邮箱成功注册
    Given 用户未登录且访问注册页面
    When 用户输入有效邮箱 "user@example.com"
    And 用户输入符合复杂度要求的密码
    And 用户点击注册按钮
    Then 系统创建未验证用户账户
    And 系统发送验证邮件至该邮箱
```

---

# Success Metrics

| 指标 | 目标值 | 测量方式 |
|:---|:---|:---|
| 用户注册转化率 | ≥ 70% | 埋点统计 |
| 核心接口 P99 延迟 | ≤ 200ms | APM 监控 |
```

### 6.3 Derive 层的特殊字段

Derive 层 frontmatter 增加追踪信息和接口摘要：

```yaml
---
doc_id: "ECOMMERCE-PLATFORM-v1.0-PAYMENT-GATEWAY-v1.0"
layer: "derive"
parent_doc: "ECOMMERCE-PLATFORM-v1.0"
parent_arch: "ECOMMERCE-PLATFORM-v1.0-ARCH"
module_name: "payment_gateway"
interfaces:
  - name: "create_payment"
    method: "POST"
    path: "/api/v1/payments"
    contract: "openapi/payment_create.yaml"
dependencies:
  - name: "order_module"
    type: "upstream"
  - name: "stripe_api"
    type: "external"
---
```

### 6.4 层级透明设计说明

`layer` 字段仅区分 `root` / `derive`，不使用 `module` / `component` 等具体层级标签。理由：

- 系统不需要知道这是"第 2 层"还是"第 3 层"
- 具体层级深度由 `parent_doc` 链隐式表达
- 避免预设层级深度带来的僵化

## 7. PRD 质量审查

在组装完成三相文档后、最终输出之前，系统必须执行**质量审查**作为内置门控。质量审查不依赖外部评审，而是流程自身的必要环节。

### 7.1 审查触发时机

质量审查在以下时刻自动触发：
- Root 模式：五大部分全部收集完毕，用户确认内容完整后
- Derive 模式：用户完成审阅修正，确认无补充后

### 7.2 SMART-REQ 检查清单

系统逐项扫描 PRD 中的每条需求条目，评估是否符合 SMART-REQ 五维标准：

| 维度 | 检验问题 | 不通过示例 | 通过示例 |
|:---|:---|:---|:---|
| **S**pecific 精确性 | 是否使用明确动作动词和具体名词？ | "系统应该很快" | "认证接口 P99 延迟 ≤ 200ms" |
| **M**easurable 可度量 | 是否包含可量化指标或可观察条件？ | "体验应该良好" | "3 秒内完成首次内容绘制" |
| **A**chievable 可实现 | 在当前约束下是否可实现？ | "支持 100 亿并发"（单实例） | "100 实例 × 100 万连接" |
| **R**elevant 相关性 | 是否直接关联上级父需求？（Derive 模式） | 与父需求无关的 nice-to-have | 每条需求标注 `parent_req` |
| **T**estable 可测试 | 是否可转化为通过/失败判定？ | "系统应具有一致性" | "并发写入同一键最终返回后完成的值" |

**检查策略：**
- **Must-Have 需求：** 五维全部强制检查，任一维度不通过即标记
- **Should/Could-Have 需求：** Specific + Testable 强制检查，其余维度警告
- **非功能需求（NFR）：** Measurable 强制检查，必须包含具体数值

**系统输出示例：**
```
SMART-REQ 检查报告
==================
✅ REQ-001: 通过全部 5 维检查
⚠️  REQ-003: Specific 不通过 — "系统应友好提示用户" 中 "友好" 为模糊量词，
   建议改为具体动作（如"显示红色错误横幅并附重试按钮"）
❌ REQ-005: Testable 不通过 — "系统应具备高可用性" 无可量化指标，
   建议补充 SLA（如"可用性 ≥ 99.9%"）
```

用户根据报告修正后重新触发检查，直至所有 Must-Have 需求通过。

### 7.3 歧义扫描

LLM 辅助进行三层歧义检测，定位为**预审工具**而非终审权威：

**第一层：词汇歧义检测**
- 扫描 PRD 中的多义词，标记建议替换术语
- 示例："用户"可能指终端用户、管理员或系统用户 → 建议明确为"终端用户"或"管理员"

**第二层：逻辑一致性验证**
- 检查需求间是否存在矛盾
- 示例：REQ-002 要求"所有 API ≤ 50ms"，REQ-008 要求"每请求完整一致性校验" → 标记潜在冲突

**第三层：完整性缺口识别**
- 对照 EARS 模式库检测是否遗漏关键类别
- 示例：检测到缺少安全性要求、缺少异常处理需求 → 提示补充

**系统输出示例：**
```
歧义扫描报告
============
🔍 词汇歧义: 发现 2 处
   - "用户"在 REQ-001、REQ-004、REQ-007 中可能指不同角色，建议统一
   - "快速处理"在 REQ-010 中为模糊量词，建议量化

🔍 逻辑一致性: 发现 1 处潜在矛盾
   - REQ-002（所有 API ≤ 50ms）与 REQ-008（完整一致性校验）可能冲突
   - 建议确认一致性校验的实现方式是否满足延迟约束

🔍 完整性缺口: 发现 1 处
   - 未检测到安全性相关需求（认证、授权、数据加密）
   - 建议补充或明确声明"安全性在后续模块中处理"
```

### 7.4 审查通过标准

| 检查项 | 通过条件 | 未通过处理 |
|:---|:---|:---|
| SMART-REQ | 所有 Must-Have 需求通过五维检查 | 逐条标记问题，用户修正后重检 |
| 歧义扫描 | 无阻塞级歧义（词汇歧义可警告通过） | 逻辑矛盾和完整性缺口必须解决 |
| Gherkin 覆盖 | 每条 Must-Have 至少 1 个场景 | 提示补充缺失场景 |
| 追踪完整性 | Derive 模式下所有需求有 `parent_req` | 自动关联或提示用户确认 |

### 7.5 质量审查与用户交互

系统以**结构化报告**形式展示审查结果，用户有三种响应方式：

1. **一键修正：** 对标记的问题，系统提供修正建议，用户确认后自动应用
2. **手动修正：** 用户自行修改 PRD 内容，修改后重新触发审查
3. **忽略并继续：** 用户声明某条警告可接受（需记录理由，如"安全性由基础设施层统一处理"）

**设计原则：** 质量审查是**协作式**而非**阻塞式**的。系统提供问题发现能力，最终判断权在用户。但所有 Must-Have 需求的 SMART 不通过项必须在输出前解决。

## 8. 边界情况处理

### 8.1 会话中断与重入

**退出条件（满足任一即结束）：**
- 用户主动输入 "完成" 或 "结束"
- 五大部分全部收集完毕且用户确认
- 用户选择 "稍后继续"

**进度保存：**
```yaml
# .prd_session_state.yaml
session_id: "sess_20260517_001"
mode: "root"
target_module: null
current_phase: "P3"
completed_phases: ["P1", "P2"]
draft_content: { ... }
parent_context: { ... }
```

用户再次启动时，系统读取状态文件，从中断阶段继续。

### 8.2 Derive 目标模块不存在

**场景：** `target_module` 在上层架构设计中不存在。

**处理：**
```
系统: "未在架构设计中找到 'payment_module'。已识别的模块列表：
      - user_service
      - order_service
      - payment_gateway
      你是否指的是 'payment_gateway'？"

用户: "是 payment_gateway" / "我需要更新架构设计"
```

- 拼写错误：自动修正后继续
- 架构未更新：提示用户先更新架构设计，或进入降级模式（基于上层 PRD 推断模块边界，标记 `tentative: true`）

### 8.3 上层 PRD 与架构设计不一致

**场景：** 上层 PRD 中某条需求涉及的功能，在上层架构设计中无对应模块。

**处理：**
1. 标记冲突项，在对话开始时向用户展示
2. 提供选项：
   - A) 假设架构遗漏，将功能作为候选模块纳入当前流程（标记 `tentative: true`）
   - B) 忽略该需求，视为未来扩展
   - C) 暂停 PRD 生成，要求先同步 PRD 与架构设计

**默认行为：** 选项 A，允许流程继续并提醒更新架构。

### 8.4 Gherkin 场景数量控制

**原则：** 覆盖优先于数量限制。每条 Must-Have 需求至少覆盖 Happy Path 和 Error Path，不设定硬性上限。

**引导策略：**
- **起步：** 引导用户为每条 Must-Have 写 1 个 Happy Path + 1 个 Error Path
- **追问边界：** 对关键需求主动追问边界条件（如"如果第三方服务超时怎么办？""并发请求下会怎样？"）
- **拆分提示：** 当单条需求的场景数 ≥ 5 且场景间差异较大时，系统提示：
  > "该需求已产生 N 个场景，是否考虑将其拆分为 2-3 条更细粒度的子需求？"

**参考指标（非强制）：**
- 简单需求（CRUD 类）：2-3 个场景
- 中等复杂度（状态流转）：3-5 个场景
- 高复杂度（核心业务流程）：5-8 个场景

> 注：参考文档中的"Gherkin 场景 ≤ 10 个"是**整份 PRD** 的停止指标之一（判断分解是否充分），而非单条需求的限制。

## 9. 设计决策记录

### ADR-001: 为什么层级只区分 root/derive？

**背景：** 传统分层方法通常显式定义项目级、模块级、组件级等层级。

**决策：** 仅使用二元标记 `root`（顶层）和 `derive`（非顶层）。

**原因：**
- 项目在启动时无法预知最终分层深度
- 模块和组件的边界在不同项目中差异巨大
- 通过 `parent_doc` 链可隐式表达完整层级路径

**后果：**
- 正面：流程更通用，不预设项目结构
- 负面：无法从单份文档直接判断"这是第几层"

### ADR-002: 为什么 Derive 模式需要上层架构设计？

**背景：** 理论上仅从上层 PRD 也可推断模块需求。

**决策：** Derive 模式强制要求上层架构设计作为输入。

**原因：**
- 上层 PRD 只定义"做什么"，不包含模块边界
- 模块的接口契约（OpenAPI、数据流）来自架构设计
- 架构设计完成后，模块列表和边界才确定

**后果：**
- 正面：模块 PRD 的接口部分精确可追踪
- 负面：Derive 模式不能在上层架构完成前启动

### ADR-003: 为什么 Root 模式采用对话而非自动生成？

**背景：** 也可以让系统直接根据用户的一句话描述生成完整 PRD。

**决策：** Root 模式采用多轮结构化对话，而非一次性自动生成。

**原因：**
- 顶层 PRD 是后续所有工作的基础，质量至关重要
- 对话过程帮助用户澄清模糊想法，发现隐含假设
- 自动生成的 PRD 容易出现幻觉需求（文档中提到的 vibe coding 失败模式）

**后果：**
- 正面：PRD 质量更高，歧义更少
- 负面：生成时间较长，需要用户投入更多注意力
