## 1. 概述

### 1.1 设计目标

本文档定义一个**可复用的 PRD（Product Requirements Document）生成流程**，服务于分层软件开发方法论。该流程的核心特性：

- **层级透明：** 除"是否为顶层"外，不区分模块级、组件级等具体层级概念
- **通用模板：** 所有层级使用同一套 PRD 模板结构（五大部分 + 三相格式）
- **独立调用：** 每次调用生成一份 PRD，两次调用之间无状态关联
- **双模式策略：**
  - **Root 模式：** 通过结构化对话引导用户，每阶段有明确的最低标准停止条件
  - **Derive 模式：** 基于上层文档**全自动**生成模块级 PRD，无人值守，适合 CI/CD 集成

### 1.2 背景

在分层软件开发中，每一层都遵循"规格 → 设计 → 验证"的元工作流循环。PRD 生成是这一循环的第一步，负责将模糊的用户意图或上层约束转化为结构化的需求规格文档。

PRD 生成流程本身**不涉及**测试用例生成、架构设计、代码实现等后续步骤，仅聚焦于产出符合标准模板的需求文档。

## 2. 核心概念

### 2.1 两种工作模式

流程在每次调用时，通过自动判定选择以下两种模式之一：

| 模式                  | 触发条件                               | 核心行为                                             | 交互策略                                                   |
| :-------------------- | :------------------------------------- | :--------------------------------------------------- | :--------------------------------------------------------- |
| **Root 模式**   | 无上层 PRD，或用户声明项目开端         | 通过多轮结构化对话完整收集 PRD 五大部分              | 每阶段系统判定最低标准，用户可随时 `done` 进入下一阶段   |
| **Derive 模式** | 提供上层 PRD + 上层架构设计 + 目标模块 | 基于上层文档**全自动**解析、预填充、拆分、输出 | **全程无交互**，质量检查自动修复，无法修复则报错退出 |

### 2.2 PRD 模板结构

所有层级统一使用以下五大部分结构：

| 部分                                 | 核心内容                                    | 撰写目标                   | 篇幅占比 |
| :----------------------------------- | :------------------------------------------ | :------------------------- | :------- |
| **元信息 Frontmatter**         | 文档 ID、版本、作者、状态、优先级、追踪链接 | 确保文档可追溯、可管理     | 5-10%    |
| **问题陈述 Problem Statement** | 目标用户、痛点描述、机会窗口                | 建立需求的合理性和紧迫性   | 10-15%   |
| **需求规格 Requirements**      | 功能/非功能需求列表、MoSCoW 优先级          | 精确描述系统行为和质量属性 | 40-50%   |
| **验收标准 Acceptance**        | Gherkin 场景、边界条件、错误路径            | 定义"完成"的客观判定标准   | 20-25%   |
| **成功指标 Success Metrics**   | 业务指标、技术指标、用户指标                | 验证需求交付后的实际价值   | 10-15%   |

### 2.3 三相文档格式

PRD 文档采用三相格式输出：

- **YAML frontmatter：** 承载结构化元数据（文档身份、状态、追踪链接、接口摘要）
- **Markdown body：** 承载人类可读的设计正文（五大部分）
- **Gherkin 场景：** 承载可执行的验收标准（Given-When-Then 格式）

## 3. 流程架构

### 3.1 入口与模式判定

流程以用户输入为起点，自动判定使用哪种模式：

```
[用户输入]
    │
    ├─ 情况 A: 自然语言描述（"我要做一个电商平台..."）
    ├─ 情况 B: 明确声明项目开端（"这是一个新项目..."）
    ├─ 情况 C: 上传上层 PRD + 架构设计 + 指定目标模块
    └─ 情况 D: 模糊输入（"帮我写个 PRD"）
    │
    ▼
[P0. 模式判定]
```

**判定规则（优先级从高到低）：**

1. **用户明确声明"项目开端" → Root 模式**（最高优先级，覆盖其他条件）
2. 同时满足以下三项 → Derive 模式：
   - 提供上层 PRD 文档
   - 提供上层架构设计文档
   - 明确指定目标模块名称
3. 以上条件不满足 → Root 模式

**情况 D 的处理：**
系统主动询问用户确认层级：

> "请确认这是项目的最顶层 PRD，还是基于已有 PRD 的下一层细化？"

## 4. Root 模式详细设计

### 4.1 适用场景

- 项目的最顶层 PRD（系统级需求）
- 用户只有一个初步想法，没有上层约束文档
- 用户明确声明"这是一个新项目/新功能的开端"

### 4.2 对话阶段与停止条件

系统按五大部分顺序引导，每个阶段有**最低标准**（系统判定）和**用户控制**（用户可随时 `done`）：

| 阶段                  | 目标                 | 最低标准（必须满足）                                                                              | 用户控制                                                                                                                                                                           |
| :-------------------- | :------------------- | :------------------------------------------------------------------------------------------------ | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P1. Frontmatter       | 收集元信息           | `doc_id`, `version`, `author`, `status`, `priority` 五个字段全部非空                    | 五个字段非空后，系统提示：`"最低要求已满足。输入 done 进入下一阶段，或补充可选字段（tags 等）："`                                                                                |
| P2. Problem Statement | 理解业务背景         | `target_users`, `pain_points`, `opportunity` 三个字段全部非空                               | 三个字段非空后，系统提示：`"背景信息已完整。输入 done 进入下一阶段，或补充更多细节："`                                                                                           |
| P3. Requirements      | 收集功能/非功能需求  | ① 至少 1 条功能需求 ② 所有功能需求都有 `priority`（Must/Should/Could） ③ 至少 1 条非功能需求 | 满足后系统提示：`"核心需求已覆盖。输入 done 进入下一阶段，或继续补充："`。若 Must-Have ≥ 3 且覆盖核心场景，额外提示 `"核心场景似乎已覆盖"`                                    |
| P4. Acceptance        | 将需求转化为 Gherkin | 每条**Must-Have** 需求至少对应 **1 个** Gherkin 场景                                  | 满足后系统提示：`"Must-Have 已全覆盖。输入 done 进入下一阶段，或补充 Error Path / 边界条件："`。若某 Must-Have 仅 1 个场景，提示 `"该需求只有 Happy Path，是否补充错误场景？"` |
| P5. Success Metrics   | 定义验收指标         | 至少**1 个**指标，且包含**具体数值**（如"≤ 200ms"而非"快速响应"）                    | 满足后系统提示：`"指标已可量化。输入 done 开始质量审查，或补充更多指标："`。用户未提供任何指标时，系统基于 P3 的 NFR 自动生成默认值                                              |

### 4.3 强制推进机制

若用户在某个阶段长时间停留（如连续跳过必填字段），系统采取渐进式干预：

1. **首次跳过必填字段：** 友好提醒 `"该字段为必填，请提供值或输入 '跳过' 使用默认值"`
2. **连续跳过 3 次：** 系统自动填充默认值并警告 `"已使用默认值：author = 'Claude'，您可在生成前修改"`
3. **用户输入 `"保存并退出"`：** 随时中断，保存会话状态，下次通过 `--resume` 恢复

### 4.4 P3. Requirements — 发散后收敛策略

**Step 1 — 发散收集：**

> "请描述这个系统应该具备哪些核心功能？用简短的一句话描述每个功能点。"

**Step 2 — 分类标注：**
系统自动应用 MoSCoW 分类，向用户确认优先级。

**Step 3 — 逐条精化（对每条 Must-Have）：**

> "关于'用户可以注册和登录'，请补充：
>
> 1. 支持哪些注册方式？
> 2. 登录后需要持久化会话吗？
> 3. 密码需要满足什么复杂度要求？"

**Step 4 — 非功能需求补充：**

> "请为整个系统定义非功能需求：并发用户数、响应时间要求、可用性目标..."

### 4.5 P4. Acceptance — Gherkin 引导生成

系统**不要求用户直接写 Gherkin**，而是引导用户描述场景，系统自动转化：

> "请描述一个用户成功注册的场景：
>
> - 用户当前处于什么状态？
> - 用户做了什么操作？
> - 系统应该有什么反应？"

系统实时转化为：

```gherkin
Scenario: 用户通过邮箱成功注册
  Given 用户未登录且访问注册页面
  When 用户输入有效邮箱 "user@example.com"
  And 用户输入符合复杂度要求的密码
  And 用户点击注册按钮
  Then 系统创建未验证用户账户
  And 系统发送验证邮件至该邮箱
```

用户确认或修改。

### 4.6 整体流程停止条件（最终）

Root 模式的**最终停止**需要同时满足：

1. **五大部分最低标准全部达成**
2. **用户明确确认生成** —— P5 结束后系统询问：
   ```
   "五大部分已收集完毕。是否现在生成 PRD？（yes / 返回修改某阶段）"
   ```
3. **质量审查通过或用户接受警告** —— SMART-REQ 和歧义扫描后，用户确认 `"接受并生成"`

用户可以在任何时候输入 `"保存并退出"` 来暂停会话，下次通过 `--resume` 恢复。

## 5. Derive 模式详细设计

### 5.1 适用场景

- 上层 PRD 和上层架构设计已完成
- 需要为架构设计中的某个具体模块自动生成 PRD
- 模块边界已由架构设计定义清楚
- **典型用途：** CI/CD 流水线中，架构设计更新后自动触发下游模块 PRD 生成

### 5.2 输入要求

| 输入项                  | 是否必须 | 说明                                       |
| :---------------------- | :------- | :----------------------------------------- |
| `parent_prd`          | 是       | 上层 PRD 文档内容或路径                    |
| `parent_architecture` | 是       | 上层架构设计文档内容或路径                 |
| `target_module`       | 是       | 目标模块名称，须存在于架构设计的模块列表中 |

### 5.3 全自动流水线

Derive 模式全程无 `input()` 调用，所有原交互点替换为**确定性自动决策规则**。

```
[输入: parent_prd + parent_architecture + target_module]
    │
    ▼
[D0. 解析上层文档]
    │    └── 提取模块上下文、相关需求、接口定义、依赖关系
    ▼
[D1. Frontmatter]
    │    └── 自动生成 doc_id / version / parent_doc / module_name
    │    └── 优先级继承上层，不询问
    ▼
[D2. Problem Statement]
    │    └── 从模块职责描述自动提炼目标用户、痛点、机会窗口
    ▼
[D3. Requirements]
    │    └── 上层相关需求按模块拆分 + 默认内部需求填充
    │    └── 自动修复 SMART 不通过项
    ▼
[D4. Acceptance]
    │    └── 每个接口自动生成 2 个场景：Happy Path + 400 Error Path
    │    └── 上层 Gherkin 场景继承并调整主语
    ▼
[D5. Success Metrics]
    │    └── 模块级默认指标自动填充（接口延迟、可用性）
    │    └── 上层指标按模块分解（如有明确分解规则）
    ▼
[质量门控]
    │    └── SMART-REQ 自动修复 → 无法修复则收集到错误报告
    │    └── 歧义扫描 → 逻辑矛盾/完整性缺口自动修复或标记
    │    └── Gherkin 覆盖检查 → 自动生成补齐缺失场景
    ▼
[组装三相文档] ──→ YAML frontmatter + Markdown body + Gherkin
    │
    ▼
[输出 PRD.md] ──→ 明确停止，返回退出码 0
```

### 5.4 自动决策规则

当前 `main.py` 的 `run_derive_mode` 中所有 `input()` 调用替换为以下规则：

| #  | 原交互                                            | 自动决策规则                                                                 |
| :- | :------------------------------------------------ | :--------------------------------------------------------------------------- |
| 1  | 模块不存在时询问 `"你是否指的是 '{similar}'？"` | 自动计算编辑距离，若最相似模块距离 ≤ 2 则自动使用；否则报错退出（退出码 1） |
| 2  | 孤儿需求处理：A/B/C 三选一                        | **默认选 A**：纳入当前流程，标记 `tentative: true`，日志输出警告     |
| 3  | `"上下文摘要如上，是否继续"`                    | 自动继续，摘要写入 `stderr` 日志                                           |
| 4  | `"是否使用上述预填充值"`（Problem Statement）   | 自动使用预填充值                                                             |
| 5  | 循环添加额外功能需求                              | **移除循环**。仅基于上层相关需求拆分，不主动扩展                       |
| 6  | 循环添加额外非功能需求                            | 保留一个默认 NFR：`"模块应符合上层架构约束"`，不询问                       |
| 7  | 质量检查失败：`"是否继续生成 PRD"`              | 触发自动修复。修复后全部通过则继续；仍有失败项则报错退出（退出码 2）         |
| 8  | Gherkin 覆盖缺口提示                              | 对每条缺少场景的 Must-Have，自动生成基础场景                                 |
| 9  | 循环添加额外 Gherkin 场景                         | **移除循环**。仅使用自动生成的场景                                     |
| 10 | 循环添加额外成功指标                              | **移除循环**。仅使用默认值 + 上层指标分解                              |
| 11 | 最终歧义扫描：`"是否继续生成最终 PRD"`          | 自动修复可修复项，无法修复的报错退出（退出码 2）                             |

### 5.5 D3. Requirements — 上层需求自动拆分规则

对每条相关上层需求，自动拆分为 **恰好 2 条**模块级需求：

```python
for req in related_requirements:
    req_id = req["id"]
    req_text = req["text"]

    # 子需求 1：接口封装层
    functional.append({
        "id": f"{req_id}-1",
        "text": f"{module_name} 应提供 {req_text} 的接口封装",
        "priority": req["priority"],  # 继承上层优先级
        "parent_req": req_id,
    })

    # 子需求 2：核心逻辑层
    functional.append({
        "id": f"{req_id}-2",
        "text": f"{module_name} 应实现 {req_text} 的核心逻辑",
        "priority": req["priority"],
        "parent_req": req_id,
    })
```

默认非功能需求（自动生成）：

```python
non_functional = [
    {"id": "NFR-001", "text": f"模块 {module_name} 应符合上层架构约束"}
]
```

### 5.6 D4. Acceptance — 接口场景自动生成规则

对每个接口自动生成 **恰好 2 个场景**：

```python
for iface in interfaces:
    iface_name = iface.get("name", "unknown")
    method = iface.get("method", "")
    path = iface.get("path", "")

    # Happy Path
    scenarios.append({
        "feature": module_name,
        "scenario": f"{iface_name} 正常调用",
        "given": f"模块 {module_name} 正常运行，依赖服务可用",
        "when": f"调用 {iface_name} 接口（{method} {path}）且参数合法",
        "then": "返回状态 200 且响应体符合接口契约",
    })

    # Error Path（400 Bad Request）
    scenarios.append({
        "feature": module_name,
        "scenario": f"{iface_name} 参数非法",
        "given": f"模块 {module_name} 正常运行",
        "when": f"调用 {iface_name} 接口且参数缺失或格式错误",
        "then": "返回状态 400 且 error_code 说明错误原因",
    })
```

### 5.7 自动修复机制

质量检查发现问题后，先尝试自动修复，无法修复才报错。

**自动修复规则库：**

| 问题类型                   | 触发条件                           | 修复规则                                   | 示例                                          |
| :------------------------- | :--------------------------------- | :----------------------------------------- | :-------------------------------------------- |
| **模糊量词**         | 需求文本含"友好"、"快速"、"大量"等 | 替换为具体动作 + 量化阈值                  | "友好提示" → "显示错误横幅并附重试按钮"      |
| **缺少量化指标**     | NFR 无具体数值                     | 按领域填充默认值                           | "高可用" → "可用性 ≥ 99.9%"                 |
| **缺少 parent_req**  | Derive 模式下需求无 `parent_req` | 文本相似度匹配，关联最相似的上层需求       | REQ-005-1 自动关联 REQ-005                    |
| **Gherkin 覆盖缺口** | Must-Have 需求无对应场景           | 按需求文本生成基础 Happy Path + Error Path | 见 D4 规则                                    |
| **歧义词汇**         | "用户"在不同需求中指不同角色       | 根据上下文推断并统一                       | 若无法推断，标记 `ambiguity: true` 但不阻塞 |
| **缺少接口契约细节** | 接口场景缺少具体请求/响应字段      | 从上层架构的 OpenAPI 定义中提取填充        | 架构无定义则使用占位符 `TODO: 补充字段`     |

**无法自动修复的问题（直接报错退出，退出码 2）：**

| 问题类型                            | 原因                          | 输出示例                                                                                |
| :---------------------------------- | :---------------------------- | :-------------------------------------------------------------------------------------- |
| **逻辑矛盾**                  | 两条需求要求互斥的行为        | `REQ-002 要求"所有 API ≤ 50ms"与 REQ-008 要求"每请求完整一致性校验"冲突，需人工裁决` |
| **目标模块不存在且无法匹配**  | 编辑距离 > 2，无相似模块      | `模块 'paymnt' 不存在，最接近的是 'payment_gateway'（距离=2），无法自动确定`          |
| **上层 PRD 与架构严重不一致** | >50% 的需求无对应模块         | `orphan 需求比例 60% 超过阈值 50%，请同步 PRD 与架构设计`                             |
| **自动修复后仍 SMART 失败**   | 修复规则无法使 Must-Have 通过 | `REQ-010: 经自动修复后仍缺少可量化指标，需人工补充`                                   |

### 5.8 退出码定义

```python
EXIT_SUCCESS = 0          # PRD 生成成功，全部质量检查通过
EXIT_INPUT_ERROR = 1      # 输入文件缺失、模块不存在、YAML 解析失败等
EXIT_QUALITY_BLOCKED = 2  # 存在无法自动修复的质量问题（附带 JSON 错误报告）
```

当退出码为 `2` 时，系统输出调试文件供人工排查：

```bash
# 成功输出
ecommerce_payment_gateway_prd_v1.0.md

# 失败输出（调试用）
ecommerce_payment_gateway_prd_v1.0.draft.md    # 未完成的内容
ecommerce_payment_gateway_prd_v1.0.errors.json  # 详细错误报告
```

### 5.9 信息隐藏原则

Derive 模式严格遵循信息隐藏原则：

- **关注当前模块内部：** 功能细节、算法逻辑、内部状态
- **关注当前模块对外契约：** 接口签名、输入输出格式（来自上层架构）
- **不关注同层其他模块：** 其他模块的内部实现与本模块 PRD 无关

## 7. PRD 质量审查

在组装完成三相文档后、最终输出之前，系统必须执行**质量审查**作为内置门控。质量审查不依赖外部评审，而是流程自身的必要环节。

### 7.1 审查触发时机

质量审查在以下时刻自动触发：

- **Root 模式：** 五大部分全部收集完毕，用户确认内容完整后
- **Derive 模式：** D5 完成后自动触发，无需用户确认

### 7.2 SMART-REQ 检查清单

系统逐项扫描 PRD 中的每条需求条目，评估是否符合 SMART-REQ 五维标准：

| 维度                        | 检验问题                                | 不通过示例                  | 通过示例                           |
| :-------------------------- | :-------------------------------------- | :-------------------------- | :--------------------------------- |
| **S**pecific 精确性   | 是否使用明确动作动词和具体名词？        | "系统应该很快"              | "认证接口 P99 延迟 ≤ 200ms"       |
| **M**easurable 可度量 | 是否包含可量化指标或可观察条件？        | "体验应该良好"              | "3 秒内完成首次内容绘制"           |
| **A**chievable 可实现 | 在当前约束下是否可实现？                | "支持 100 亿并发"（单实例） | "100 实例 × 100 万连接"           |
| **R**elevant 相关性   | 是否直接关联上级父需求？（Derive 模式） | 与父需求无关的 nice-to-have | 每条需求标注 `parent_req`        |
| **T**estable 可测试   | 是否可转化为通过/失败判定？             | "系统应具有一致性"          | "并发写入同一键最终返回后完成的值" |

**检查策略：**

- **Must-Have 需求：** 五维全部强制检查，任一维度不通过即标记
- **Should/Could-Have 需求：** Specific + Testable 强制检查，其余维度警告
- **非功能需求（NFR）：** Measurable 强制检查，必须包含具体数值

**Derive 模式自动修复：**

Derive 模式下，SMART 检查不通过时先执行自动修复（见 5.7 节），修复后重新检查。自动修复后仍不通过的 Must-Have 需求触发 `EXIT_QUALITY_BLOCKED`（退出码 2）。

**Root 模式用户交互：**

Root 模式下，系统输出检查报告，用户有三种响应方式：

1. **一键修正：** 对标记的问题，系统提供修正建议，用户确认后自动应用
2. **手动修正：** 用户自行修改 PRD 内容，修改后重新触发审查
3. **忽略并继续：** 用户声明某条警告可接受（需记录理由，如"安全性由基础设施层统一处理"）

### 7.3 歧义扫描

LLM 辅助进行三层歧义检测，定位为**预审工具**而非终审权威：

**第一层：词汇歧义检测**

- 扫描 PRD 中的多义词，标记建议替换术语
- 示例："用户"可能指终端用户、管理员或系统用户 → 建议明确为"终端用户"或"管理员"

**第二层：逻辑一致性验证**

- 检查需求间是否存在矛盾
- 示例：REQ-002 要求"所有 API ≤ 50ms"，REQ-008 要求"每请求完整一致性校验" → 标记潜在冲突

**第三层：完整性缺口识别**

- 对照 EARS 模式库检测是否遗漏关键类别
- 示例：检测到缺少安全性要求、缺少异常处理需求 → 提示补充

**Derive 模式处理：**

- **逻辑矛盾：** 无法自动修复，直接报错退出（退出码 2）
- **完整性缺口：** 尝试自动补充默认项（如缺少安全性需求则添加 `"模块应遵循上层安全策略"`），若无法推断则报错
- **词汇歧义：** 自动推断统一，推断失败则标记 `ambiguity: true` 但不阻塞

**Root 模式输出示例：**

```
歧义扫描报告
============
🔍 词汇歧义: 发现 2 处
   - "用户"在 REQ-001、REQ-004、REQ-007 中可能指不同角色，建议统一
   - "快速处理"在 REQ-010 中为模糊量词，建议量化

🔍 逻辑一致性: 发现 1 处潜在矛盾
   - REQ-002（所有 API ≤ 50ms）与 REQ-008（完整一致性校验）可能冲突
   - 建议确认一致性校验的实现方式是否满足延迟约束

🔍 完整性缺口: 发现 1 处
   - 未检测到安全性相关需求（认证、授权、数据加密）
   - 建议补充或明确声明"安全性在后续模块中处理"
```

### 7.4 审查通过标准

| 检查项       | 通过条件                               | 未通过处理                                                           |
| :----------- | :------------------------------------- | :------------------------------------------------------------------- |
| SMART-REQ    | 所有 Must-Have 需求通过五维检查        | Root: 逐条标记问题，用户修正后重检；Derive: 自动修复 → 仍失败则报错 |
| 歧义扫描     | 无阻塞级歧义（词汇歧义可警告通过）     | Root: 逻辑矛盾和完整性缺口必须解决；Derive: 自动修复 → 仍失败则报错 |
| Gherkin 覆盖 | 每条 Must-Have 至少 1 个场景           | Root: 提示补充缺失场景；Derive: 自动生成补齐                         |
| 追踪完整性   | Derive 模式下所有需求有 `parent_req` | Derive: 自动关联或报错                                               |

### 7.5 质量审查与交互模式对比

| 维度         | Root 模式              | Derive 模式                  |
| :----------- | :--------------------- | :--------------------------- |
| 触发时机     | 用户确认五大部分完成后 | D5 完成后自动触发            |
| SMART 失败   | 报告用户，由用户决策   | 自动修复 → 修复失败报错     |
| 歧义发现     | 报告用户，由用户决策   | 自动修复或标记 → 阻塞级报错 |
| Gherkin 缺口 | 提示用户补充           | 自动生成补齐                 |
| 最终决策权   | 用户                   | 系统（退出码决定）           |

## 8. 边界情况处理

### 8.1 会话中断与重入（Root 模式）

**阶段内停止：**

- 用户输入 `"保存并退出"` —— 保存当前会话状态，退出程序
- 用户输入 `"done"` —— 当前阶段最低标准已满足，进入下一阶段

**最终停止（满足任一即结束）：**

- 五大部分全部收集完毕，质量审查通过，用户确认生成
- 用户主动输入 `"保存并退出"`（可恢复）

**进度保存：**

```yaml
# .prd_session_state.yaml
session_id: "sess_20260517_001"
mode: "root"
target_module: null
current_phase: "P3"
completed_phases: ["P1", "P2"]
draft_content: { ... }
parent_context: { ... }
```

用户再次启动时，系统读取状态文件，从中断阶段继续。

### 8.2 Derive 目标模块不存在

**场景：** `target_module` 在上层架构设计中不存在。

**全自动处理：**

1. 计算 `target_module` 与所有可用模块的编辑距离
2. 若最小距离 ≤ 2：自动使用最相似模块，日志输出警告
3. 若最小距离 > 2：报错退出（退出码 1），输出可用模块列表到 `stderr`

```
# 距离 ≤ 2，自动修正
[WARNING] 模块 'paymnt' 不存在，自动使用最接近的 'payment_gateway'（编辑距离=2）

# 距离 > 2，报错退出
[ERROR] 模块 'xyz_service' 不存在。可用模块:
  - user_service
  - order_service
  - payment_gateway
```

### 8.3 上层 PRD 与架构设计不一致

**场景：** 上层 PRD 中某条需求涉及的功能，在上层架构设计中无对应模块。

**Derive 模式处理：**

1. 计算 orphan 需求比例
2. 若比例 ≤ 50%：标记冲突项为 `tentative: true`，纳入当前流程，日志输出警告
3. 若比例 > 50%：报错退出（退出码 2），提示先同步 PRD 与架构设计

**Root 模式处理：**

1. 标记冲突项，在对话开始时向用户展示
2. 提供选项：
   - A) 假设架构遗漏，将功能作为候选模块纳入当前流程（标记 `tentative: true`）
   - B) 忽略该需求，视为未来扩展
   - C) 暂停 PRD 生成，要求先同步 PRD 与架构设计
3. **默认行为：** 选项 A，允许流程继续并提醒更新架构。

### 8.4 Gherkin 场景数量控制

**原则：** 覆盖优先于数量限制。每条 Must-Have 需求至少覆盖 Happy Path 和 Error Path，不设定硬性上限。

**Root 模式引导策略：**

- **起步：** 引导用户为每条 Must-Have 写 1 个 Happy Path + 1 个 Error Path
- **追问边界：** 对关键需求主动追问边界条件（如"如果第三方服务超时怎么办？""并发请求下会怎样？"）
- **拆分提示：** 当单条需求的场景数 ≥ 5 且场景间差异较大时，系统提示：
  > "该需求已产生 N 个场景，是否考虑将其拆分为 2-3 条更细粒度的子需求？"
  >

**Derive 模式自动生成策略：**

- 每个接口恰好生成 2 个场景（Happy Path + 400 Error）
- 上层 Gherkin 场景继承并调整主语
- 不主动追问边界条件，避免过度生成

**参考指标（非强制）：**

- 简单需求（CRUD 类）：2-3 个场景
- 中等复杂度（状态流转）：3-5 个场景
- 高复杂度（核心业务流程）：5-8 个场景

> 注：参考文档中的"Gherkin 场景 ≤ 10 个"是**整份 PRD** 的停止指标之一（判断分解是否充分），而非单条需求的限制。

## 9. 设计决策记录

### ADR-001: 为什么层级只区分 root/derive？

**背景：** 传统分层方法通常显式定义项目级、模块级、组件级等层级。

**决策：** 仅使用二元标记 `root`（顶层）和 `derive`（非顶层）。

**原因：**

- 项目在启动时无法预知最终分层深度
- 模块和组件的边界在不同项目中差异巨大
- 通过 `parent_doc` 链可隐式表达完整层级路径

**后果：**

- 正面：流程更通用，不预设项目结构
- 负面：无法从单份文档直接判断"这是第几层"

### ADR-002: 为什么 Derive 模式需要上层架构设计？

**背景：** 理论上仅从上层 PRD 也可推断模块需求。

**决策：** Derive 模式强制要求上层架构设计作为输入。

**原因：**

- 上层 PRD 只定义"做什么"，不包含模块边界
- 模块的接口契约（OpenAPI、数据流）来自架构设计
- 架构设计完成后，模块列表和边界才确定

**后果：**

- 正面：模块 PRD 的接口部分精确可追踪
- 负面：Derive 模式不能在上层架构完成前启动

### ADR-003: 为什么 Root 模式采用对话而非自动生成？

**背景：** 也可以让系统直接根据用户的一句话描述生成完整 PRD。

**决策：** Root 模式采用多轮结构化对话，而非一次性自动生成。

**原因：**

- 顶层 PRD 是后续所有工作的基础，质量至关重要
- 对话过程帮助用户澄清模糊想法，发现隐含假设
- 自动生成的 PRD 容易出现幻觉需求（文档中提到的 vibe coding 失败模式）

**后果：**

- 正面：PRD 质量更高，歧义更少
- 负面：生成时间较长，需要用户投入更多注意力

### ADR-004: 为什么 Derive 模式改为全自动无交互？

**背景：** v1.0 中 Derive 模式为混合交互（预填充 + 用户审阅修正）。

**决策：** v2.0 将 Derive 模式重构为全自动无人值守，全程无 `input()`。

**原因：**

- Derive 模式的核心用途是 CI/CD 集成，流水线中不允许人工干预
- 模块级 PRD 的输入（上层 PRD + 架构设计）已经足够结构化，机器可以确定性处理
- 自动修复规则可以持续积累，形成项目的"领域知识库"

**后果：**

- 正面：可集成到 CI/CD，架构更新后自动触发下游模块 PRD 生成
- 负面：无法处理需要人类判断的模糊场景（通过退出码 2 和错误报告暴露给人工）

### ADR-005: 为什么 Root 模式采用"最低标准 + 用户控制"混合策略？

**背景：** 纯用户主导容易无限发散；纯系统主导体验生硬。

**决策：** 系统判定每阶段最低标准，达到后建议推进，但用户始终可以 `done` 或继续补充。

**原因：**

- 最低标准防止用户在单阶段无限停留（如反复修改 Frontmatter 而不进入正题）
- 用户保留控制权，避免系统过于强势
- 不同项目、不同用户的"足够好"标准不同，无法统一

**后果：**

- 正面：平衡了效率与灵活性
- 负面：最低标准的阈值需要经验调优（初期可能偏松或偏紧）
